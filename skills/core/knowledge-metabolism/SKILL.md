# Knowledge Metabolism

## Route the material before changing doctrine

Classify the input as raw capture, source snapshot, synthesis, candidate mechanism, experiment result, skill patch candidate, or standing knowledge. Keep ephemeral work in chat, durable source truth in the vault, repeatable procedures in skills, deterministic transforms in scripts, and structured recurring state in ledgers.

Capture is not promotion. Move material only as far as its evidence and use justify:

```text
raw capture -> routed source -> candidate synthesis -> tested application -> validated artifact -> canonical knowledge
```

At any stage, merge, challenge, demote, archive, or reject it.

## Revise append-first

When new evidence contradicts a durable belief, revise the belief artifact itself with an append — never overwrite or replace it, and never leave the revision only in prose:

1. Preserve the previous claim, date, evidence, confidence, and scope.
2. Append the new observation with its source handle.
3. Explain whether the update changes the claim, confidence, boundary, or application.
4. Name the concrete `decision_delta`—the decision, artifact, rule, or future extraction behavior that changes. A revision that names a `decision_delta` is `act` (the changed application is next), not `done`.
5. Keep unresolved disagreement visible instead of forcing a false reconciliation.

A single correction normally updates a hypothesis. Repeated evidence or one explicit, authorized standing instruction may change doctrine.

## Apply the promotion gate

Before calling material canonical, require provenance, scope, confidence, strongest surviving objection, rival explanation, falsifier, failure modes, non-application boundary, observed use, demotion trigger, review date, and rollback or merge path. Missing fields keep the item speculative or candidate; status labels do not create truth.

For a proposed skill or runbook change, also require a named target, recurring evidence, expected future behavior change, duplicate-coverage check, minimal patch, safety/privacy check, and deterministic or human validation gate. Record rejected patches so the same weak proposal does not recur.

## Review for application rent

Retain promoted knowledge only while it improves retrieval, decisions, artifacts, or future behavior enough to justify context and review cost. At review, strengthen, narrow, merge, demote, supersede, or kill it. Preserve provenance even when active prominence is removed.
